● Summary of commit 16439f419b270184ec501c531bf20d83b6745fb0                    
                                                                                
  This commit removes SVID error handling from the fast path of fmod/fmodf to   
  improve performance.                                                          
                                                                                
  Background: SVID Math ABI                                                     
                                                                                
  SVID (System V Interface Definition) is a legacy error handling mechanism     
  where math functions call a user-definable matherr() callback on errors. This 
  is obsolete but glibc maintains it for binary compatibility.                  
                                                                                
  What Changed                                                                  
                                                                                
  1. New implementation in e_fmod.c: The core function now handles errors       
  directly using IEEE-style error handling (__math_edom sets errno = EDOM)      
  instead of going through the SVID wrapper.                                    
  2. Compatibility symbols preserved: For binaries linked against glibc < 2.38, 
  the old __fmod_compat wrapper with SVID handling is kept via compat_symbol(). 
  New binaries (glibc 2.38+) get the faster version.                            
  3. Symbol versioning: New fmod symbol added at GLIBC_2.38 in all the          
  libm.abilist files.                                                           
  4. Arch-specific handling:                                                    
    - ia64: unchanged (uses its own __libm_error_region)                        
    - i686/m68k: get wrappers to avoid needing new symbols                      
                                                                                
  Key Code Change                                                               
                                                                                
  // Before: always went through SVID wrapper                                   
  // After: inline error handling, no matherr() call                            
  if (__glibc_unlikely (hy == 0 || hx >= EXPONENT_MASK || hy > EXPONENT_MASK))  
  {                                                                             
    if (is_nan (hx) || is_nan (hy))                                             
      return (x * y) / (x * y);        // NaN propagation                       
    return __math_edom ((x * y) / (x * y));  // Set EDOM, return NaN            
  }                                                                             
                                                                                
  Performance Gains                                                             
                                                                                
  The removal of the wrapper overhead gives 10-30% speedup on subnormals and    
  close-exponent cases across x86_64, aarch64, and armhf.

❯ Suppose I have a program written for fmod@@GLIBC_2.38 and expecting its       
  semantics. I tweak it so it finds the old fmod@GLIBC_2.0 implementation       
  instead. What goes wrong (if anything) and what conditions are necessary      
  for things to go wrong? Be concise but precise.                               
                                                                                
● What goes wrong:                                                              
                                                                                
  The old fmod@GLIBC_2.0 may invoke matherr() on domain errors, while the new   
  version never does. If a matherr() function exists (user-defined or from      
  another library):                                                             
  1. It gets called unexpectedly                                                
  2. If it returns non-zero, errno is not set (breaking code that checks errno  
  == EDOM)                                                                      
  3. If it modifies exc.retval, you get a different return value instead of NaN 
                                                                                
  Conditions required:                                                          
                                                                                
  All must be true:                                                             
  1. Domain error occurs: isinf(x) || y == 0.0 with neither argument being NaN  
  2. _LIB_VERSION != _IEEE_: True by default (typically _POSIX_ or _SVID_)      
  3. A matherr() symbol is linked: For actual behavioral divergence beyond the  
  call itself                                                                   
                                                                                
  If no custom matherr() exists, the observable behavior is identical (NaN      
  returned, errno = EDOM). The difference is only the code path taken and       
  potential side effects from an interposed matherr().

❯ Is this true of _all_ changes in glibc regarding SVID error handling,         
  that if we are guaranteed no matherr symbol is present in the running         
  program, there is no observable difference between the two ABIs?              
                                                                                
● No, that's not universally true. Even without matherr(), there are observable 
  differences:                                                                  
                                                                                
  1. _SVID_ mode prints to stderr                                               
                                                                                
  When _LIB_VERSION == _SVID_, the old path through __kernel_standard prints    
  error messages like "fmod: DOMAIN error\n" to stderr. The new path never does.
                                                                                
  2. Return values can differ for some error types                              
                                                                                
  SVID specifies particular return values that differ from IEEE 754:            
  ┌────────────────┬─────────────┬─────────────┐                                
  │   Error type   │ SVID return │ IEEE return │                                
  ├────────────────┼─────────────┼─────────────┤                                
  │ log(-1) DOMAIN │ -HUGE_VAL   │ NaN         │                                
  ├────────────────┼─────────────┼─────────────┤                                
  │ pow(0,-1) SING │ ±HUGE_VAL   │ ±inf        │                                
  ├────────────────┼─────────────┼─────────────┤                                
  │ Overflow       │ ±HUGE       │ ±inf        │                                
  └────────────────┴─────────────┴─────────────┘                                
  The new implementations return IEEE 754 values unconditionally.               
                                                                                
  3. errno edge cases                                                           
                                                                                
  SVID distinguishes SING (singularity, e.g., 1/0) from DOMAIN. Both map to EDOM
   but the logic paths differ. Most new implementations are careful here, but   
  the mapping isn't guaranteed identical for all functions.                     
                                                                                
  ---                                                                           
  For fmod specifically, the only semantic difference without matherr is the    
  missing stderr message in _SVID_ mode, since its error case (DOMAIN) already  
  returns NaN in both ABIs.                                                     
                                                                                
  For functions with overflow/singularity behavior (pow, log, exp, lgamma,      
  etc.), the return value differences are real and observable even with         
  _LIB_VERSION == _POSIX_ in some cases, depending on how faithfully each       
  function's new implementation preserved the old wrapper's return value logic.

  etc.) uses IEEE 754                                                           
                                                                                
  The HUGE_VAL abstraction exists because:                                      
                                                                                
  1. C standard portability - ISO C was designed to work on non-IEEE-754 systems
   (VAX, IBM System/360 hexadecimal float, Cray, etc.)                          
  2. Historical baggage - These systems existed when the C standard was written 
  3. Header conformance - glibc's <math.h> must conform to the C standard's     
  wording                                                                       
                                                                                
  But in glibc's actual code, HUGE_VAL is unconditionally __builtin_huge_val()  
  (which GCC defines as +inf), or the fallback 1e10000 (which overflows to +inf 
  on IEEE 754).                                                                 
                                                                                
  So my earlier claim holds: in glibc, HUGE_VAL == +inf always, because glibc   
  doesn't support any non-IEEE-754 floating-point system. The abstraction is    
  purely for standards conformance, not practical portability.
