  ---
  Differences between exp@GLIBC_2.2.5 and exp@@GLIBC_2.29

  The underlying math is identical. Both call the same __ieee754_exp
  implementation from sysdeps/ieee754/dbl-64/e_exp.c. The split was introduced
  in commit c20a10561a ("Remove the error handling wrapper from exp and exp2",
  2018).

  The differences are entirely in the wrapper logic:

  exp@GLIBC_2.2.5 (math/w_exp_compat.c)

  double z = __ieee754_exp(x);
  if (!isfinite(z) || z == 0)
      if (isfinite(x) && _LIB_VERSION != _IEEE_)
          return __kernel_standard(x, x, 6 + !!signbit(x));
  return z;

  exp@@GLIBC_2.29 (math/w_exp_template.c)

  double z = __ieee754_exp(x);
  if (!isfinite(z) || z == 0)
      if (isfinite(x))
          __set_errno(ERANGE);
  return z;

  ---
  So, is SVID error handling the only difference? Not quite — there are three 
  distinct behavioral differences:

  ┌────────────────────┬────────────────────────────┬──────────────────────┐
  │      Scenario      │      exp@GLIBC_2.2.5       │   exp@@GLIBC_2.29    │
  ├────────────────────┼────────────────────────────┼──────────────────────┤
  │ Overflow,          │                            │ Returns ∞, errno =   │
  │ _LIB_VERSION ==    │ Returns ∞, no errno set    │ ERANGE               │
  │ _IEEE_             │                            │                      │
  ├────────────────────┼────────────────────────────┼──────────────────────┤
  │ Overflow,          │                            │ Returns ∞, errno =   │
  │ _LIB_VERSION ==    │ Returns ∞, errno = ERANGE  │ ERANGE ✓ same        │
  │ _POSIX_            │                            │                      │
  ├────────────────────┼────────────────────────────┼──────────────────────┤
  │ Overflow,          │ Returns HUGE (3.40e+38f,   │ Returns ∞, errno =   │
  │ _LIB_VERSION ==    │ finite!), calls matherr()  │ ERANGE               │
  │ _SVID_             │                            │                      │
  ├────────────────────┼────────────────────────────┼──────────────────────┤
  │ Underflow,         │                            │ Returns 0, errno =   │
  │ _LIB_VERSION ==    │ Returns 0, calls matherr() │ ERANGE               │
  │ _SVID_             │                            │                      │
  └────────────────────┴────────────────────────────┴──────────────────────┘

  The three concrete differences:

  1. matherr() support: The old version calls __kernel_standard which invokes a
  user-defined matherr() hook in SVID/XOPEN modes. The new version never calls
  matherr().
  2. Return value for overflow in _SVID_ mode: The old version returns HUGE
  (defined as 3.40282347e+38F in math/math-svid-compat.h — a finite float value,
   not infinity). The new version always returns the raw __ieee754_exp result
  (infinity). This is a different return value, not merely a different
  error-reporting path.
  3. errno in _IEEE_ mode: The old version skips all error reporting when
  _LIB_VERSION == _IEEE_, so errno is never set. The new version unconditionally
   sets errno = ERANGE for overflow/underflow even in IEEE mode.

  For programs using the default POSIX mode (_LIB_VERSION == _POSIX_), the two
  versions are behaviorally identical. The differences only matter for programs
  that explicitly set _LIB_VERSION or define matherr() — which is very old,
  non-portable code.

  The Versions file comment (# No SVID compatible error handling.) is accurate
  but undersells point 2: in _SVID_ mode, the old version returns a different 
  value for overflow, not just different error plumbing.

● Here's the write-up:

  ---
  What We Proved About glibc Math Symbol Version Changes

  Background

  glibc uses symbol versioning to maintain ABI compatibility. Several math
  functions have two exported versions in libm.so.6: an old compat symbol (e.g.
  exp@GLIBC_2.2.5) and a newer default symbol (e.g. exp@@GLIBC_2.29). The
  question was whether these differ only in SVID error handling, or whether
  there are other changes a caller in default POSIX mode would observe.

  ---
  What We Established

  1. Source-level analysis of the wrapper structure (thorough, authoritative)

  Starting from exp as a representative case and reading through the source and
  git history (key commit: c20a10561a, "Remove the error handling wrapper from
  exp and exp2", 2018), we established the general pattern:

  - The old symbol is a thin wrapper (w_*_compat.c) that calls the underlying
  implementation and then, if the result is exceptional (overflow, underflow,
  domain error), calls __kernel_standard() — the SVID error machinery.
  - The new symbol is a different thin wrapper (or sometimes the implementation
  itself with a new versioned_symbol declaration) that calls the same underlying
   implementation and directly sets errno.

  The differences between old and new in any mode are:

  ┌───────────────────────┬──────────────────────────┬──────────────────────┐
  │       Behavior        │      Old @GLIBC_2.x      │   New @@GLIBC_2.y    │
  ├───────────────────────┼──────────────────────────┼──────────────────────┤
  │ matherr() callback    │ Called (in SVID/XOPEN    │ Never called         │
  │                       │ mode)                    │                      │
  ├───────────────────────┼──────────────────────────┼──────────────────────┤
  │ _LIB_VERSION ==       │ Suppresses all error     │ Ignored; errno       │
  │ _IEEE_                │ reporting                │ always set           │
  ├───────────────────────┼──────────────────────────┼──────────────────────┤
  │ Return value on       │ Returns HUGE (~3.4e38,   │                      │
  │ overflow in _SVID_    │ finite)                  │ Returns +∞           │
  │ mode                  │                          │                      │
  ├───────────────────────┼──────────────────────────┼──────────────────────┤
  │ Return value on       │                          │ Returns +∞ —         │
  │ overflow in _POSIX_   │ Returns +∞               │ identical            │
  │ mode                  │                          │                      │
  ├───────────────────────┼──────────────────────────┼──────────────────────┤
  │ errno in _POSIX_ mode │ Set to ERANGE/EDOM via   │ Set to ERANGE/EDOM   │
  │                       │ __kernel_standard        │ directly — identical │
  └───────────────────────┴──────────────────────────┴──────────────────────┘

  In _POSIX_ mode (the glibc default for any program compiled with standard 
  feature-test macros), return values and errno are identical between old and 
  new. The underlying math code — the actual floating-point computation — is the
   same in both cases.

  2. Binary analysis across all version-bumped symbols (analyze_math_symbols.py)

  The script disassembles both the old and new symbol bodies from a compiled
  libm.so.6 and checks whether they branch to the same underlying implementation
   address. On the x86_64 system tested, all 13 math functions with SVID-related
   version bumps show Same fn? = YES:

  exp, exp2, exp10, exp10f, log, pow, hypot, hypotf,
  fmod, fmodf, lgamma, lgammaf, lgammal

  This confirms that the binary matches the source analysis: the wrappers
  differ, the implementation does not.

  3. Runtime behavioral test in POSIX mode (test_math_compat.c)

  The C test loads both symbol versions via dlvsym, asserts _LIB_VERSION == 
  _POSIX_, then calls both with ~34–44 inputs per function covering: normal
  values, overflow triggers, underflow triggers, domain-error triggers (log(-x),
   pow(-x, 0.5)), subnormals, ±0, ±∞, and NaN. It compares return values
  bit-exactly (with NaN == NaN) and errno.

  Results on x86_64: all tests pass for exp, exp2, log, pow, hypot, fmod, 
  lgamma.

  ---
  What This Does NOT Prove

  1. Other architectures. The runtime test was only run on x86_64. The binary
  analysis script supports cross-arch use (set OBJDUMP/READELF to cross-tools
  and pass the target's libm.so.6), but has not been run on other arches. The
  version string for the old symbol varies by arch (GLIBC_2.2.5 on x86_64,
  GLIBC_2.0 on i386, GLIBC_2.17 on aarch64, etc.).
  2. totalorder / totalordermag are different. These functions version-bumped
  from GLIBC_2.25/2.26/2.27 to GLIBC_2.31 not to remove SVID handling, but
  because their argument types changed from values to pointers. Old and new are
  not call-compatible. The binary analysis confirms the same underlying
  implementation is reached, but the calling convention is different and the
  test program's type signatures don't apply to them.
  3. Floating-point exception flags. The test compares return values and errno
  only. It does not check whether the same FP exception flags (fegetexcept) are
  raised by both versions. This is expected to be identical since both call the
  same implementation, but was not explicitly verified.
  4. _IEEE_ and _SVID_ mode callers. The test only covers _POSIX_ mode. Callers
  that explicitly set _LIB_VERSION = _IEEE_ will observe a difference (old: no
  errno; new: errno set). Callers that set _LIB_VERSION = _SVID_ will observe a
  difference in matherr() callback behavior and in the overflow return value
  (HUGE vs +∞). These differences are intentional and documented in the
  math/Versions comment # No SVID compatible error handling.
  5. Exhaustive input coverage. The runtime test uses a fixed set of ~34–44
  inputs per function. It does not exhaustively cover all 2^64 doubles. The
  boundary inputs are chosen to stress the error-handling paths specifically,
  but a fuzzer-style test was not run.
  6. Only the currently installed libm.so.6 was tested. Other glibc versions may
   have different internal structures. The analysis applies to whatever binary
  you run the scripts against.
